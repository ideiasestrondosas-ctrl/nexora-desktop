# Plano de Análise — Nexora Desktop v0.18.0

## 1. Visão Geral da Aplicação

A aplicação está num estado **muito maduro** para uma v0.18.0:
- **Arquitetura sólida**: Tauri 2.x + React 19 + Sidecar Node.js, SQLite local, IPC nativo
- **Pipeline completo**: 8 workers (ingest → QC → transcode → audio → proxy → thumbnail → QC pós → delivery)
- **Multi-plataforma**: Windows/macOS/Linux com builds funcionais
- **Qualidade de código**: TypeScript strict, testes, i18n (15 idiomas), lint/format

No entanto, existem **oportunidades claras** nas três áreas que pediste: **velocidade/processamento**, **interface gráfica**, e **usabilidade para leigos**.

---

## 2. Performance & Velocidade de Processamento

### Problemas Identificados

| # | Problema | Impacto | Severidade |
|---|---|---|---|
| P1 | **Polling IPC agressivo** — Dashboard 10s, Queue 2s, Library 5s, System metrics 2s. Milhares de chamadas Rust↔JS por hora. | CPU/bateria desnecessárias, possível "jank" na UI | Alta |
| P2 | **TopBar GPU gauge é fake** — sempre mostra 15% ou 0%, não lê utilização real da GPU | Métrica enganosa, desperdício de render | Média |
| P3 | **Transcode sem paralelização real** — o sidecar parece processar 1 job de cada vez; o `max_concurrent_jobs` (1-4) não está a ser respeitado no pipeline | Velocidade total limitada a 1 FFmpeg mesmo com CPU/RAM disponíveis | Alta |
| P4 | **SHA-256 no ingest sem progresso** — ficheiros de 10GB+ bloqueiam o worker sem feedback | Experiência de "app congelada" | Média |
| P5 | **Mutex único na BD Rust** — `Mutex<Connection>` serializa TODAS as queries SQLite; sem pool de conexões | Contenção sob carga, queries bloqueiam umas às outras | Média |
| P6 | **Recharts carregado globalmente** — biblioteca de gráficos pesada (~70kB gzip) só usada no Dashboard, mas presente no bundle principal | Bundle maior, parse mais lento | Baixa |
| P7 | **Library grid sem virtualização** — vista em grelha renderiza TODOS os assets de uma vez (só a lista é virtualizada) | Lag com +100 assets, memória excessiva | Média |

### Recomendações de Otimização

1. **Substituir polling por eventos push (Tauri `Emitter`)**
   - O Rust já emite `system-metrics` e `disk-space`. Estender o mesmo padrão para `jobs:updated`, `assets:updated`, `queue:changed`.
   - Eliminar `setInterval(fetchData, ...)` no Dashboard, Library e Queue. Só fazer fetch quando o Rust notificar.

2. **Paralelização de jobs no sidecar**
   - Implementar worker pool no sidecar (ex: `p-limit` ou `worker_threads`) para respeitar `max_concurrent_jobs`.
   - Cada job independente pode correr o seu FFmpeg em paralelo — a única limitação deve ser CPU/RAM/disco.

3. **Pool de conexões SQLite no Rust**
   - Substituir `Mutex<Connection>` por `r2d2_sqlite` ou `deadpool-sqlite` para queries concorrentes.
   - Ou, alternativamente, usar `tokio::sync::RwLock` se manter a arquitetura síncrona.

4. **Lazy-loading do Recharts**
   - `React.lazy(() => import('recharts'))` no Dashboard para split do bundle de gráficos.

5. **Virtualização da grelha da Library**
   - Usar `@tanstack/react-virtual` também para a vista em grelha (grid virtualization), não só para lista.

6. **GPU gauge real ou remoção**
   - Se não for possível ler utilização real da GPU (especialmente em macOS/Linux), substituir por um simples indicador "GPU ativa/inativa" ou remover o gauge.

## 3. Interface Gráfica (UI/UX)

### Problemas Identificados

| # | Problema | Impacto |
|---|---|---|
| U1 | **Terminologia excessivamente técnica** em toda a UI: "VMAF", "LUFS", "QC", "NVENC", "libx264", "True Peak (dBTP)", "Container" | Interface intimidadora para não-técnicos |
| U2 | **Pipeline de 8 passos visível** — mostra "ingest → qc-pre → transcode → audio → proxy → thumbnail → qc-post → delivery" com nomes em inglês misturados | Informação em excesso; o utilizador comum só quer saber "a processar..." |
| U3 | **SettingsPage é uma "painel de controlo de avião"** — 5 tabs, 20+ opções técnicas, sliders de VMAF/LUFS que um leigo não entende | Ansiedade cognitiva, medo de mexer |
| U4 | **Badge VMAF no library card** — "VMAF 93.2" não comunica nada a quem não sabe o que é VMAF | Informação inútil para 90% dos utilizadores |
| U5 | **TopBar com 4 gauges circulares** ocupa ~40% da largura com métricas de sistema | Poluição visual; o utilizador comum não precisa de ver CPU/RAM em tempo real |
| U6 | **Botão "Play" na Library não funciona** — `<button className="..."><Play /></button>` sem `onClick` | UI quebrada, frustração |
| U7 | **Paginação da Library está mockada** — botões "Anterior/Próximo" sempre desabilitados/habilitados sem funcionalidade real | Indicador de funcionalidade incompleta |
| U8 | **Cores e contrastes** — em dark mode, o fundo `#0a0d14` com texto `#f3f4f6` é bom, mas alguns elementos `text-text-muted` (`#6b7280`) em dark podem não passar WCAG AA em monitores com brilho baixo | Acessibilidade |

### Recomendações de UI

1. **Modo Simples vs Modo Avançado**
   - Introduzir um toggle global (ou detecção automática baseada no primeiro uso) que filtra a UI:
     - **Simples**: esconde gauges de sistema, esconde pipeline detalhado, mostra só "A preparar → A converter → A finalizar", esconde VMAF/LUFS nas cards, esconde tabs técnicas nas Settings.
     - **Avançado**: UI actual para profissionais de vídeo.

2. **Redesenhar o QueuePage para leigos**
   - Em vez de 8 passos com nomes técnicos, mostrar **3 fases visuais** com ícones grandes:
     1. "A analisar o vídeo" (ícone de lupa)
     2. "A converter" (ícone de engrenagem giratória)
     3. "A guardar" (ícone de check)
   - O pipeline técnico de 8 passos pode ficar num tooltip ou modo expandido.

3. **Simplificar SettingsPage**
   - **Modo Simples** só mostra:
     - Pasta de saída
     - Tema / Idioma
     - "Usar aceleração GPU" (toggle)
     - "Número de vídeos ao mesmo tempo" (1-4)
   - Esconder: VMAF threshold, LUFS, True Peak, CPU preset, H.264 profile, telemetria.

4. **Substituir badges técnicos por indicadores visuais**
   - Em vez de "VMAF 93.2", mostrar um **selo de qualidade**: "Qualidade Excelente" + ícone verde.
   - Em vez de "codec: h264", mostrar um ícone de compatibilidade: "Compatível com tudo".

5. **Compactar a TopBar**
   - Remover os 4 gauges circulares. Substituir por:
     - Nome do ecrã + descrição
     - Um indicador discreto: "1 vídeo a processar" (quando ativo)
     - Botões de ajuda e saída

6. **Corrigir funcionalidade quebrada**
   - Implementar `onClick` no botão Play da Library (abrir no player nativo do SO via Tauri `opener`).
   - Implementar paginação real ou remover os botões mockados.

# Plano de Análise Detalhada e Completa — Nexora Desktop

Este documento apresenta uma análise profunda do estado atual da aplicação **Nexora Desktop** (arquitetura Tauri + React + Node.js Sidecar), focando-se em três pilares fundamentais solicitados: **Melhorias Gerais**, **Otimização de Velocidade/Processamento** e **Reformulação de Interface/UX para Utilizadores Leigos**.

> [!IMPORTANT]
> **Nenhuma alteração no código foi realizada.** Este é exclusivamente o plano de análise e ação. Aguardo a sua aprovação ou feedback sobre os pontos levantados antes de avançar para qualquer modificação.

---

## 1. Análise Arquitetural e Estado Atual

A aplicação utiliza uma stack moderna e robusta:
*   **Frontend**: React 19, Vite, TailwindCSS v4, Zustand (estado), componentes Radix UI e ícones Lucide.
*   **Backend (Tauri/Rust)**: Gere o ciclo de vida da app, base de dados (SQLite), métricas de sistema (CPU/RAM/Rede) e a fila de processamento (queue).
*   **Processamento de Vídeo (Sidecar Node.js)**: Executa tarefas (jobs) delegando para o FFmpeg/FFprobe de forma "stateless" (um processo por tarefa).

---

## 2. Otimizações de Velocidade e Processamento

### Backend (Rust / Tauri)
*   **Concorrência Assíncrona**: Atualmente, métricas de disco e sistema utilizam `std::thread::spawn` com bloqueios `std::thread::sleep`. 
    *   *Proposta*: Migrar para **Tokio** (`tokio::spawn` e `tokio::time::sleep`), o que reduz o consumo de recursos (overhead de threads do SO) e torna a execução em background mais leve e eficiente.
*   **Otimização da Base de Dados (SQLite)**: 
    *   *Proposta*: Ativar o modo **WAL (Write-Ahead Logging)** e `PRAGMA synchronous = NORMAL` no `db.rs`. Isto vai acelerar substancialmente as leituras e escritas simultâneas (muito frequentes com logs e progresso de jobs).

### Motor de Processamento (Sidecar Node.js / FFmpeg)
*   **Aceleração de Hardware (GPU)**: O processamento de vídeo pode ser brutalmente lento no CPU.
    *   *Proposta*: Integrar deteção automática de GPU (NVENC para NVIDIA, QSV para Intel, VideoToolbox para Mac) e ajustar dinamicamente os argumentos do FFmpeg para usar a placa gráfica, reduzindo os tempos de processamento de horas para minutos.
*   **Gestão de Recursos do Sidecar**: O Node.js tem um tempo de arranque (startup time) associado.
    *   *Proposta*: Para processamento em lote (batch), otimizar o sidecar para não ter de iniciar o ambiente Node para cada pequeno vídeo, reduzindo a sobrecarga em batch jobs de muitos ficheiros pequenos.
*   **Processamento Paralelo**: Permitir ao utilizador configurar (ou auto-ajustar baseado nos cores do CPU) o processamento de X vídeos em simultâneo na fila (Queue).

### Frontend (React)
*   **Lazy Loading e Code Splitting**: O `App.tsx` carrega todas as páginas (Dashboard, Library, Settings, etc.) de uma só vez.
    *   *Proposta*: Usar `React.lazy` e `Suspense` para carregar as tabs da interface apenas quando o utilizador clica nelas. Isto reduz drasticamente o tempo de arranque (First Contentful Paint) da UI.
*   **Virtualização de Listas**: Se a "Library" e os "Logs" tiverem centenas de itens, a interface vai engasgar.
    *   *Proposta*: Assegurar que componentes de lista longa usam virtualização (já vi que existe o `@tanstack/react-virtual` no `package.json`, precisamos garantir que está implementado corretamente nas páginas de listagem).

## 3. Melhoria da Interface Gráfica (UI) e Experiência do Utilizador (UX) para Leigos

O maior desafio atual da aplicação é a sua "linguagem técnica". Para um utilizador normal/leigo, a aplicação deve ser o mais simples, limpa e guiada possível.

### Simplificação de Terminologia
*   **"Ingest Asset"** $\rightarrow$ **"Adicionar Vídeo"** ou **"Importar"**. A palavra "Ingest" (Ingerir) é jargão técnico de broadcast/IT.
*   **"Profiles"** $\rightarrow$ **"Predefinições de Qualidade"** ou **"Formatos"**.
*   Em vez de perguntar por *Codecs* e *Bitrates* logo de início, oferecer opções baseadas em intenção:
    *   *"Alta Qualidade (Ideal para arquivo)"*
    *   *"Poupar Espaço (Reduzir tamanho para WhatsApp/Email)"*
    *   *"Formato Web (Otimizado para YouTube/Tiktok)"*
    *   *(E esconder as definições avançadas de codec/resolução num botão "Avançado")*

### Reformulação do Fluxo Principal (Onboarding & Ações)
1.  **Empty States (Ecrãs Vazios Amigáveis)**: Quando o utilizador abre a app e não há vídeos, mostrar um ícone grande, acolhedor, com uma mensagem simples: "Arraste os seus vídeos para aqui para começar" em vez de apenas uma lista vazia ou métricas a zero.
2.  **Redução de Alertas Técnicos Intimidantes**:
    *   *Atualmente*: No arranque, se o Node.js ou FFmpeg falharem, o utilizador recebe um aviso: *"Instale o Node.js 20+ no PATH"*. Um leigo **não sabe** o que é Node.js nem o que é PATH.
    *   *Proposta UX*: A aplicação deve tentar fazer download automático das dependências (binários do FFmpeg e Node estático) de forma invisível. Caso não consiga, o alerta deve ser um ecrã de boas vindas que diz *"Estamos a preparar as ferramentas necessárias..."* com um botão de "Reparar Automático".
3.  **Progresso Humano**:
    *   Em vez de apenas mostrar percentagens e "ETA" ou logs técnicos, mostrar mensagens como: *"A preparar o vídeo...", "A reduzir o tamanho...", "Quase pronto!"*.
    *   Mostrar o tempo restante de forma orgânica: *"Cerca de 2 minutos restantes"*.

### Melhorias Visuais (UI Estética)
*   **Cores e Contraste**: O tema atual já tem fundos neutros (`#f8fafc`, `#0a0d14`), mas podemos aplicar um design mais moderno (tipo Apple/Vercel) com "Glassmorphism" (transparências subtis e blur) nos modais e na TopBar.
*   **Micro-interações e Animações**: Adicionar transições suaves entre tabs e ao adicionar vídeos à lista (usando bibliotecas como `framer-motion` ou CSS nativo otimizado). Isto dá à app um aspeto "Premium" e responsivo.
*   **Sidebar Simplificada**: O menu lateral pode ser colapsável ou apenas de ícones (com tooltips) para dar mais espaço de ecrã ao conteúdo de vídeo, que é o foco principal.
