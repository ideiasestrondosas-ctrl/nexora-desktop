# v0.23.0 — UX Fixes Design Spec

**Data:** 2026-05-18  
**Autor:** ideiasestrondosas-ctrl  
**Versão:** 1.0

---

## 1. Contexto e Motivação

Cinco problemas de UX identificados após a release v0.22.0:

1. No detalhe do asset, o link "abrir ficheiro processado" no histórico de jobs usa `openPath()` (abre com aplicação do SO) mas o utilizador quer navegação dentro da app.
2. O histórico de jobs no detalhe do asset mostra todos os jobs do sistema em vez de apenas os do asset em visualização.
3. No menu Fila → separadores "Concluídos" e "Histórico", o popup de reprocessamento aparece atrás de outros elementos da tabela (problema de overflow/clipping).
4. No Resumo do Pipeline, os ícones de contagem por fase são apenas informativos; o utilizador quer clicar neles para ver quais ficheiros estão em cada estado e navegar para o detalhe de cada um.
5. Quando se elimina um asset ou se efectua factory reset, o utilizador quer controlo explícito sobre se os ficheiros processados/temporários no disco devem também ser eliminados.

---

## 2. Âmbito

Todas as alterações são em `dev` branch, versão v0.23.0. Sem novas páginas. Sem alterações à estrutura da base de dados.

## 3. Item 1 — Job History: "Abrir ficheiro processado" navega in-app

### Problema

O botão usa `openPath(job.output_path)` do `@tauri-apps/plugin-opener`. Abre o ficheiro com a aplicação padrão do SO. O utilizador quer permanecer dentro da app e navegar para o asset de output (se existir na biblioteca).

### Solução

**Rust — novo comando `find_asset_by_path`**

Ficheiro: `src-tauri/src/commands/assets.rs`

```rust
#[tauri::command]
pub async fn find_asset_by_path(
    path: String,
    state: tauri::State<'_, AppState>,
) -> Result<Option<String>, String> {
    let db = state.db.lock().map_err(|e| e.to_string())?;
    let result = db
        .query_row(
            "SELECT id FROM assets WHERE source_path = ? LIMIT 1",
            rusqlite::params![path],
            |row| row.get::<_, String>(0),
        )
        .optional()
        .map_err(|e| e.to_string())?;
    Ok(result)
}
```

Registar em `src-tauri/src/lib.rs` no array de comandos.

**Frontend — `AssetDetailPage`**

- Nova prop: `onSelectAsset?: (id: string) => void`
- `App.tsx` passa `handleSelectAsset` para `AssetDetailPage`
- Lógica do botão:

```tsx
async function handleOpenProcessedFile(outputPath: string) {
  try {
    const assetId = await invoke<string | null>('find_asset_by_path', {
      path: outputPath,
    });
    if (assetId && onSelectAsset) {
      onSelectAsset(assetId);
      return;
    }
  } catch {
    // continuar para fallback
  }
  await revealItemInDir(outputPath).catch(() => {});
}
```

- O `revealItemInDir` (do `@tauri-apps/plugin-opener`) é o fallback que abre a pasta e destaca o ficheiro, caso o output não esteja na biblioteca.

### Critérios de aceitação

- Clicar "abrir ficheiro processado" quando o output_path corresponde a um asset na biblioteca → navega para o detalhe desse asset
- Quando não existe correspondência → revela o ficheiro no explorador do sistema
- Sem erros visíveis ao utilizador em nenhum dos casos

## 4. Item 2 — Job History: filtrar por asset actual

### Problema

`AssetDetailPage` chama `invoke<Job[]>('list_jobs', { asset_id: assetId })` mas o handler Rust em `src-tauri/src/commands/jobs.rs` pode não aplicar o filtro `WHERE asset_id = ?`, mostrando todos os jobs.

### Solução

Verificar a query SQL no handler `list_jobs`. Se não existir o filtro, corrigir:

```sql
-- Antes (possível bug):
SELECT * FROM jobs ORDER BY created_at DESC LIMIT ?

-- Depois (correcto):
SELECT * FROM jobs WHERE asset_id = ? ORDER BY created_at DESC
```

O parâmetro `asset_id` enviado pelo frontend (`{ asset_id: assetId }`) deve ser mapeado para o parâmetro Rust e usado na query.

### Critérios de aceitação

- No detalhe do asset A, o histórico mostra apenas jobs onde `asset_id = A.id`
- Verificar com dois assets diferentes que tenham cada um o seu histórico de jobs distinto

## 5. Item 3 — Reprocess popup em foreground (QueuePage)

### Problema

O popover de reprocessamento em `src/pages/QueuePage.tsx` está dentro de um `<div className="relative">` dentro de uma linha de tabela. A tabela ou um ancestral tem `overflow: hidden` ou `overflow: auto`, recortando o popover e fazendo-o aparecer atrás de outros elementos.

### Solução

Usar `createPortal` do React para renderizar o popover no `document.body`, fora da hierarquia da tabela. Posicioná-lo com coordenadas absolutas calculadas via `getBoundingClientRect()` do botão trigger.

```tsx
import { createPortal } from 'react-dom';

// Ref para o botão trigger
const reprocessBtnRef = useRef<HTMLButtonElement | null>(null);
const [popoverPos, setPopoverPos] = useState<{ top: number; right: number } | null>(null);

function openReprocessPopover(jobId: string, btn: HTMLButtonElement) {
  const rect = btn.getBoundingClientRect();
  setPopoverPos({
    top: rect.top + window.scrollY,
    right: window.innerWidth - rect.right,
  });
  setReprocessPopover(jobId);
}

// Popover via portal:
{
  reprocessPopover &&
    popoverPos &&
    createPortal(
      <div
        style={{
          position: 'fixed',
          top: popoverPos.top,
          right: popoverPos.right,
          zIndex: 9999,
        }}
        data-reprocess-popover
      >
        {/* conteúdo do popover */}
      </div>,
      document.body,
    );
}
```

O handler de click-outside existente (que verifica `data-reprocess-popover`) continua a funcionar.

### Critérios de aceitação

- O popup de reprocessamento aparece sempre sobre todos os outros elementos da tabela
- Fecha ao clicar fora (comportamento existente mantido)
- Funciona em todas as linhas da tabela, incluindo as do fundo da lista

## 6. Item 4 — Pipeline Summary: painel expansível inline (Opção C)

### Problema

Os badges de contagem em `src/components/PipelineSummary.tsx` (ex: "3 ✓", "1 ✗") são display-only. O utilizador quer clicar para ver que ficheiros estão nesse estado e navegar para o detalhe de cada um.

### Design escolhido — Opção C: painel expansível inline

Clicar num badge expande um sub-painel directamente abaixo da linha da fase, sem modal nem popover externo.

### Alterações em `PipelineSummary.tsx`

**Interface `Job` alargada:**

```typescript
interface Job {
  id: string;
  asset_id: string; // novo — para navegação
  profile: string; // novo — exibir no painel
  filename: string | null; // novo — exibir nome do ficheiro
  status: string;
  step: string | null;
  progress: number;
}
```

**Nova prop:**

```typescript
interface PipelineSummaryProps {
  jobs: Job[];
  onSelectAsset?: (id: string) => void; // novo
}
```

**Estado de expansão:**

```typescript
const [expandedKey, setExpandedKey] = useState<string | null>(null);
// key = `${phase}-${statusFilter}`, ex: "transcode-done"
```

**Lógica de toggle:**

```typescript
function toggleExpand(phase: string, filter: string) {
  const key = `${phase}-${filter}`;
  setExpandedKey((prev) => (prev === key ? null : key));
}
```

**Filtro dos jobs para o painel:**

```typescript
function getJobsForPhase(phase: string, filter: string): Job[] {
  return jobs.filter((j) => {
    const matchesPhase = j.step === phase;
    if (filter === 'done') return matchesPhase && j.status === 'done';
    if (filter === 'error') return matchesPhase && j.status === 'error';
    if (filter === 'active') return matchesPhase && j.status === 'processing';
    return false;
  });
}
```

**Renderização do painel inline** (abaixo de cada linha de fase, quando `expandedKey === key`):

```tsx
{
  expandedKey === key && (
    <div className="border-t border-border bg-bg-secondary">
      {filteredJobs.map((job) => (
        <div
          key={job.id}
          className="flex items-center gap-3 px-4 py-2 text-xs hover:bg-bg-tertiary"
        >
          <StatusIcon status={job.status} />
          <span className="flex-1 truncate font-mono text-text-primary">
            {job.filename ?? job.id}
          </span>
          <span className="text-text-muted shrink-0">{job.profile}</span>
          {onSelectAsset && (
            <button
              onClick={() => onSelectAsset(job.asset_id)}
              className="text-brand hover:underline shrink-0"
            >
              →
            </button>
          )}
        </div>
      ))}
      {filteredJobs.length === 0 && (
        <div className="px-4 py-2 text-xs text-text-muted">
          {t('queue.pipeline.noFilesForStatus')}
        </div>
      )}
    </div>
  );
}
```

### Alterações em `QueuePage.tsx`

- Passar `filename` e `asset_id` e `profile` nos jobs fornecidos a `<PipelineSummary jobs={...} />`
- Adicionar prop `onSelectAsset={onSelectAsset}` ao componente

### Critérios de aceitação

- Clicar no badge "3 ✓" da fase Transcode → expande painel com 3 ficheiros
- Clicar no mesmo badge novamente → fecha o painel
- Clicar num badge diferente → fecha o anterior e abre o novo
- Clicar na seta "→" de um ficheiro → navega para o detalhe do asset
- Se não houver ficheiros para esse estado → mensagem "sem ficheiros"

## 7. Item 5 — Eliminar com autorização explícita para ficheiros no disco

### Problema

As operações de eliminação existentes já deletam ficheiros do disco, mas a confirmação é única e genérica. O utilizador quer controlo explícito e separado sobre a eliminação de ficheiros processados/temporários.

### Solução A — Por asset (`delete_asset`)

**Fluxo de dois passos:**

1. Dialog existente: "Apagar asset? Esta acção é irreversível." → \[Cancelar\] \[Apagar\]
2. Se confirmado, segundo dialog: "Eliminar também os ficheiros processados do disco?" → \[Não, manter ficheiros\] \[Sim, eliminar\]

O Rust recebe `delete_files: bool` como parâmetro:

```rust
#[tauri::command]
pub async fn delete_asset(
    id: String,
    delete_files: bool,
    state: tauri::State<'_, AppState>,
) -> Result<(), String> {
    // ... lógica existente de cancelamento de jobs ...
    // só elimina ficheiros do disco se delete_files == true
    if delete_files {
        for path in &output_paths {
            let _ = std::fs::remove_file(path);
        }
    }
    Ok(())
}
```

Frontend (`AssetDetailPage.tsx` e `LibraryPage.tsx`): adicionar segundo `dialog.confirm` após o primeiro e passar `delete_files` na chamada `invoke`.

### Solução B — Factory Reset

Acrescentar checkbox "Eliminar também os ficheiros de output do disco" (marcada por omissão) à dialog de confirmação existente em `SettingsPage.tsx`.

```tsx
const [deleteFiles, setDeleteFiles] = useState(true);

// Na dialog (usar modal customizado em vez de window.confirm nativo):
<label className="flex items-center gap-2 text-sm">
  <input type="checkbox" checked={deleteFiles} onChange={(e) => setDeleteFiles(e.target.checked)} />
  {t('settings.advanced.factoryResetDeleteFiles')}
</label>;
```

O Rust `factory_reset` recebe `delete_files: bool`:

```rust
#[tauri::command]
pub async fn factory_reset(
    delete_files: bool,
    app: tauri::AppHandle,
    state: tauri::State<'_, AppState>,
) -> Result<(), String> {
    // ... lógica existente de limpeza da DB ...
    if delete_files {
        // eliminar directório temp e ficheiros de output
    }
    app.restart();
    Ok(())
}
```

### Critérios de aceitação

- Apagar asset → dois dialogs sequenciais; escolher "Não" no segundo mantém os ficheiros no disco mas remove o asset da biblioteca
- Factory reset → checkbox visível na dialog; desmarcar → reset da DB sem toque nos ficheiros do disco
- Em ambos os casos, os ficheiros de source original nunca são eliminados
